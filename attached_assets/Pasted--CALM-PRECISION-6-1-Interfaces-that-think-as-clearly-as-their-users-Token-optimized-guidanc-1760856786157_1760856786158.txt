## CORE PRINCIPLES

### 1. Group, Don’t Isolate

**Gestalt Proximity + Common Region**  
Single border around related items, dividers between them. Individual borders imply separation.

### 2. Size = Importance

**Fitts’ Law**  
Button size matches user intent weight. Critical conversions = large, quick actions = compact.

### 3. Three-Line Hierarchy

**Cognitive Load + Information Scent**  
Title (14-16px, bold) → Description (12-14px) → Metadata (11-12px, muted). Consistent vertical rhythm.

### 4. Progressive Disclosure

**Hick’s Law**  
Show less, reveal more on demand. Fewer visible choices = faster decisions.

### 5. Text Over Decoration

**Signal-to-Noise Ratio**  
Color and weight create hierarchy, not boxes. Remove decoration that doesn’t aid comprehension.

### 6. Content Over Chrome

**Information Density**  
≥70% content-chrome ratio. Users come for content, not interface.

### 7. Natural Language

**Mental Models**  
Readable phrases over jargon. Match user vocabulary.

### 8. Rhythm & Alignment

**Gestalt Continuity**  
8pt grid, consistent spacing, aligned baselines. Visual rhythm creates calm.

### 9. Functional Integrity

**Affordance Theory + Data Integrity**  
Every interactive element must have:

1. A working action (UI behavior)
1. Real data source/destination (backend connection)

**DON’T BUILD:**

- Buttons without backend endpoints
- Forms without submission APIs
- Lists without data sources
- Features as “visual placeholders”
- Mock data that looks like real data

**IF BUILDING INCREMENTALLY:**

- Mark clearly what’s functional vs mock
- Hide incomplete features rather than showing broken ones
- Document which data is real vs placeholder
- Use visible “Demo mode” warnings

**WHY:** Non-functional interfaces violate user trust. Mock data creates false mental models. Placeholders waste cognitive effort.

### 10. Content Resilience

**Fault Tolerance**  
Components handle variable formats gracefully: structured data, plain text, markdown.

## PATTERN CATALOG

### Three-Line Content Structure

**Title:** 14-16px, medium weight, high contrast (7:1)  
**Description:** 12-14px, regular, medium contrast (4.5:1), 3-4 line limit  
**Metadata:** 11-12px, regular, low contrast (3:1)

**Spacing:** 3-4px after title, 4px after description (8pt grid aligned)

**Mobile:** Single line preferred, max 2 lines for title

**Applies to:** Search results, list items, cards, table rows

### Grouped Container

```
SECTION HEADER (11px uppercase, gray-500)
┌─────────────────────────────────┐ ← Single border
│ Title            │    Metadata  │
│ Description      │              │
├─────────────────────────────────┤ ← Gradient divider
│ Title 2          │    Metadata  │
│ Description      │              │
└─────────────────────────────────┘
```

**When to group:** Items share type, user scans multiple, actions apply to group

**When NOT:** Items distinct, only 1-2 items, full-screen single item focus

**Divider options:**

- Subtle 1px line (default)
- Gradient fade (modern, softer visual separation)
- No divider if spacing sufficient (≥24px)

### Flexible Content Handling

**Accept:**

- Object with varying field names
- Plain text (single/multi-paragraph)
- Markdown (headings, bold, bullets, links, paragraphs)
- Null/undefined (show empty state)

**Field name alternatives (try in order):**

- Title: `title` → `headline` → `name` → `subject`
- Description: `description` → `summary` → `content` → `body`
- Date: `date` → `timestamp` → `published` → `created_at`

**Minimum markdown support (all content components):**

- Headings: `#` `##` `###`
- Bold: `**text**`
- Bullets: `- item`
- Paragraphs: Double newline

**Layout rules:**

- Headings: Single line with truncate + tooltip in cards
- Body: Natural wrap, leading-relaxed (1.5-1.6)
- Alignment: Top-align multi-column content
- Length: 3-4 lines visible, expand on demand

### Status Indication

**Text color only** (no background boxes):

- Success: `green-600`
- Warning: `amber-600`
- Info: `blue-600`
- Error: `red-600`
- Neutral: `gray-600`

**Exception:** Removable filter chips need subtle background (tap target)

**Why:** Background adds visual weight without information value

### Navigation States

**Top Nav (Fiverr-style):**

- Unselected: `gray-600` regular
- Hover: `gray-900` color only (150ms)
- Selected: `gray-900` medium + 2px bottom border
- Never: Background boxes or pills

**Side Nav (Google Classroom-style):**

- Unselected: `gray-700` + `hover:bg-gray-100`
- Selected: `bg-blue-50 text-blue-700` medium weight
- Icons: 18px same color as text
- Width: 224px fixed
- Padding: 12px horizontal, 8px vertical

**Section headers:** 11px uppercase, `gray-500`, tracking-wide

**Hierarchical (Folk-style):**

- Indent nested: 12px
- Chevron rotates 90° (150ms)
- Indicators: 8×8px squares
- Badges: End of row, minimal

**Dropdowns:**

- Width: 224px (matches side nav)
- Padding: 16px horizontal
- Dividers: 1px `gray-100`, 4px margin
- Hover: `bg-gray-50`
- Danger: Separated + `red-600` + `hover:bg-red-50`

### Element State Patterns

**HIDE:** Permanently unavailable (lacks permission)

- Don’t show in DOM
- Example: Admin features for non-admins

**DISABLE:** Contextually unavailable (user can fix)

- `bg-gray-200 text-gray-400 cursor-not-allowed`
- Optional helper text explaining condition
- Example: Save button until form valid

**MARK:** Available but requires action/upgrade

- Normal styling + badge
- “Coming soon” = amber badge
- “Pro” = purple badge, links to upgrade
- “Beta” = blue badge
- Example: Premium features, upcoming features

### Empty States

**Not:** “No results found”

**Instead:** Value-driven guidance

- Recent searches (clickable)
- Common searches
- Contextual suggestions
- Clear next action

**Pattern:** Optional icon → purpose statement → CTA

### Loading States

**<100ms:** No indicator (feels instant)  
**100ms-1s:** Spinner or pulse  
**1s-3s:** Skeleton screen (matches final layout)  
**>3s:** Progress bar (if determinate)

**Skeleton rules:**

- Match three-line pattern for lists
- Subtle shimmer (1.5s cycle)
- Max 5 skeleton items
- Progressive loading: header → content → secondary (50-100ms stagger)

## IMPLEMENTATION GUIDE (FOR LLMs)

### Auto-Apply Rules

1. **Content structure** → Three-line pattern (title, description, metadata)
1. **Grouping** → Single border + dividers for related items
1. **Button sizing** → Core conversion = full, quick action = compact
1. **Category headers** → Above grouped content (11px uppercase gray-500)
1. **Search debounce** → 300ms
1. **Touch targets** → ≥44px mobile, ≥24px desktop
1. **Loading states** → Match wait time (skeleton for 1-3s)
1. **Empty states** → Provide value (recent/suggestions)
1. **Status** → Text color only
1. **Icons** → Max 2 colors per context
1. **Content input** → Accept string OR object
1. **Field names** → Try alternatives (title/headline/name)
1. **Markdown** → Support `#` `**` `-` paragraphs minimum
1. **Functional integrity** → Only build with real backend endpoints
1. **Data sources** → Verify API exists before building UI

### Always Ask Before

1. Individual borders on list items (violates grouping)
1. Full-width buttons for non-critical actions (violates Fitts)
1. Background boxes for status (violates signal-to-noise)
1. 2 icon colors in same context (violates cognitive load)
1. Technical jargon without confirmation (violates mental models)
1. Making non-functional elements look interactive (violates affordance)
1. **Creating UI without real data source/API endpoint**
1. **Building forms without backend submission handler**
1. **Showing features with mock data that looks real**

### Flexible Zones (User Can Override)

**Maintain principle, vary implementation:**

- Color values (maintain contrast ratios)
- Border radius (stay consistent)
- Font family (maintain size/weight ratios)
- Spacing values (stay on 8pt grid)

**Strict zones (audit as violations):**

- Group vs individual borders
- Button sizing by context
- Three-line hierarchy
- Touch targets ≥44px mobile
- Contrast ≥4.5:1 text, ≥3:1 large
- Content-chrome ≥70%
- Functional integrity (no fake buttons)
- **Real data sources required**

### Content Handling Pattern

When building components:

```javascript
// Accept flexible input
function ContentCard({ data }) {
  // Handle different types
  if (typeof data === 'string') {
    return renderMarkdown(data);
  }
  
  // Extract with flexible field names
  const title = data.title || data.headline || data.name;
  const desc = data.description || data.summary || data.content;
  const meta = data.date || data.timestamp;
  
  // Render what exists, omit what doesn't
  return (
    <div>
      {title && <h3 className="font-medium">{title}</h3>}
      {desc && <p className="text-gray-600">{desc}</p>}
      {meta && <span className="text-gray-500 text-xs">{meta}</span>}
    </div>
  );
}
```

When generating content (in prompts):

```
Use markdown for rich formatting:
**Heading text**

Body paragraph with natural flow.

- Bullet point one
- Bullet point two

*Metadata info*
```

### Functional Integrity Implementation

**CRITICAL: Verify backend before building UI**

```jsx
// ❌ DON'T: Mock data that looks real
const [items, setItems] = useState([]);
<button onClick={() => setItems([...items, { id: Math.random(), name: 'New' }])}>
  Save
</button>

// ❌ DON'T: Placeholder API calls
<button onClick={() => console.log('TODO: implement save')}>
  Save
</button>

// ✅ DO: Only build if backend exists
{hasBackendAPI && (
  <button onClick={async () => {
    const result = await fetch('/api/save', {
      method: 'POST',
      body: JSON.stringify(formData)
    });
    // Handle real response
  }}>
    Save
  </button>
)}

// ✅ DO: Mark clearly as demo if prototyping
<div className="border-2 border-amber-500 rounded-lg p-4">
  <div className="flex items-center gap-2 mb-2">
    <AlertTriangle size={16} className="text-amber-700" />
    <p className="text-amber-700 text-xs font-medium">
      Demo mode - not connected to real data
    </p>
  </div>
  <button onClick={handleMockSave} className="...">
    Save (Demo Only)
  </button>
</div>

// ✅ DO: Hide incomplete features
{isFeatureComplete && (
  <button onClick={handleRealSave}>Save</button>
)}
```

**When building incrementally:**

1. Check if backend API exists
1. If no → Don’t build UI yet OR clearly mark as demo
1. If yes → Verify API works before building
1. Document which features are real vs mock

## ACCESSIBILITY (WCAG 2.2 AA)

### Non-Negotiable Minimums

|Requirement      |Minimum                        |Why                  |
|-----------------|-------------------------------|---------------------|
|Text contrast    |4.5:1 normal, 3:1 large        |Low vision legibility|
|Touch targets    |44×44px mobile, 24×24px desktop|Fitts’ Law for input |
|Focus indicators |Visible on all interactive     |Keyboard navigation  |
|Color + text/icon|Never color alone              |Colorblind support   |
|Motion respect   |Honor prefers-reduced-motion   |Vestibular disorder  |

### Quick Accessibility Checks

- [ ] Keyboard navigate entire UI?
- [ ] Focus visible on all interactive?
- [ ] Information clear without color?
- [ ] Touch targets meet minimums?
- [ ] Animations stop when user prefers reduced motion?

## SEMANTIC TOKENS

**Purpose:** User customizes while maintaining perceptual logic

```
Contrast (always relative, ratios enforced):
  contrast.high    → ~7:1   → main text
  contrast.medium  → ≥4.5:1 → descriptions  
  contrast.low     → ≥3:1   → metadata
  contrast.accent  → ≥4.5:1 → links, selected

Surface:
  surface.base      → page background
  surface.elevated  → cards, raised elements
  surface.grouped   → container for list groups

Border:
  border.group    → outer group border
  border.divider  → dividers within groups
  border.subtle   → hairline separators
```
