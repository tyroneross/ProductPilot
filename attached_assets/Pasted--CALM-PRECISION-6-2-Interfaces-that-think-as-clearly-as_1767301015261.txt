# CALM PRECISION 6.2

*Interfaces that think as clearly as their users*

> **Token-optimized guidance for LLM implementation**  
> Plain language • Decision trees • Perceptual foundations • Audit-ready

## PHILOSOPHY

**Goal:** Cognitive predictability through information-first structure within accessibility envelope

**Method:** Encode decision trees that LLMs can interpret and audit

**Foundation:** Each rule traces to perceptual science (Gestalt, Fitts, Hick, Cognitive Load, Signal-to-Noise)

-----

## CORE PRINCIPLES

### 1. Group, Don’t Isolate

**Gestalt Proximity + Common Region**  
Single border around related items, dividers between them. Individual borders imply separation.

### 2. Size = Importance

**Fitts’ Law**  
Button size matches user intent weight. Critical conversions = large, quick actions = compact.

### 3. Three-Line Hierarchy

**Cognitive Load + Information Scent**  
Title (14-16px, bold) → Description (12-14px) → Metadata (11-12px, muted). Consistent vertical rhythm.

### 4. Progressive Disclosure

**Hick’s Law**  
Show less, reveal more on demand. Fewer visible choices = faster decisions.

### 5. Text Over Decoration

**Signal-to-Noise Ratio**  
Color and weight create hierarchy, not boxes. Remove decoration that doesn’t aid comprehension.

### 6. Content Over Chrome

**Information Density**  
≥70% content-chrome ratio. Users come for content, not interface.

### 7. Natural Language

**Mental Models**  
Readable phrases over jargon. Match user vocabulary.

### 8. Rhythm & Alignment

**Gestalt Continuity**  
8pt grid, consistent spacing, aligned baselines. Visual rhythm creates calm.

### 9. Functional Integrity

**Affordance Theory + Data Integrity**  
Every interactive element must have:

1. A working action (UI behavior)
1. Real data source/destination (backend connection)

**DON’T BUILD:**

- Buttons without backend endpoints
- Forms without submission APIs
- Lists without data sources
- Features as “visual placeholders”
- Mock data that looks like real data

**IF BUILDING INCREMENTALLY:**

- Mark clearly what’s functional vs mock
- Hide incomplete features rather than showing broken ones
- Document which data is real vs placeholder
- Use visible “Demo mode” warnings

**WHY:** Non-functional interfaces violate user trust. Mock data creates false mental models. Placeholders waste cognitive effort.

### 10. Content Resilience

**Fault Tolerance**  
Components handle variable formats gracefully: structured data, plain text, markdown.

### 11. Mobile-First Structure *(New in 6.2)*

**Responsive Design**  
Base styles target mobile. Breakpoints add complexity for larger screens. Touch targets sized by screen position.

## DECISION TREES

### Border Usage

```mermaid
graph TD
    A[Need visual grouping?] -->|Yes| B{Items share type?}
    A -->|No| C[Use whitespace only]
    B -->|Yes| D[Single border around ALL<br/>Dividers between items]
    B -->|No| E[Category headers<br/>Separate groups]
```

**Audit:** Do borders group or isolate?

### Button Sizing

```mermaid
graph TD
    A[User intent?] --> B{Core conversion?}
    B -->|Yes| C[Full width<br/>e.g. login, checkout]
    B -->|No| D{Equal choices?}
    D -->|Yes| E[Side-by-side equal<br/>e.g. yes/no, join/skip]
    D -->|No| F[Compact inline<br/>e.g. save, edit, cancel]
```

**Audit:** Does size match intent weight?

### Content Format

```mermaid
graph TD
    A[Receive content] --> B{Type?}
    B -->|Object| C[Try flexible field names<br/>title/headline/name]
    B -->|String with markdown| D[Render markdown]
    B -->|Plain string| E[Render with paragraph breaks]
    B -->|Null/undefined| F[Show empty state]
    
    C --> G[Apply three-line hierarchy]
    D --> G
    E --> G
```

**Audit:** Does component handle string AND object inputs?

### Functional Integrity Check

```mermaid
graph TD
    A[Building interactive element?] --> B{Has backend API?}
    B -->|Yes| C{API implemented?}
    B -->|No| D[STOP - Don't build yet]
    C -->|Yes| E[Build with real data]
    C -->|No| F{Demo/prototype only?}
    F -->|Yes| G[Mark clearly as demo<br/>Don't look production-ready]
    F -->|No| D
```

**Audit:** Is this connected to real backend or mock?

### Element State

```mermaid
graph TD
    A[Has working action?] --> B{Yes}
    A --> C{No, but...}
    B --> D[Make interactive]
    C --> E{Coming soon?}
    C --> F{Needs upgrade?}
    C --> G{Permanent no access?}
    E --> H[Mark 'Coming soon'<br/>or hide]
    F --> I[Mark 'Pro'<br/>link to upgrade]
    G --> J[HIDE entirely]
```

**Audit:** If I click this, will something happen?

### Progressive Disclosure

```mermaid
graph TD
    A[Content type?] --> B{Essential?}
    B -->|Yes| C[Always visible]
    B -->|No| D{Frequently needed?}
    D -->|Yes| E[Expand on hover/tap]
    D -->|No| F[Behind action/link]
```

**Audit:** Is user forced to see unnecessary detail?

### Loading State

```mermaid
graph TD
    A[Expected wait?] --> B{<100ms}
    B -->|Yes| C[No indicator]
    B -->|No| D{100ms-1s}
    D -->|Yes| E[Spinner/pulse]
    D -->|No| F{1s-3s}
    F -->|Yes| G[Skeleton screen]
    F -->|No| H[Progress bar]
```

**Audit:** Does loading pattern match wait time?

### Touch Target Sizing *(New in 6.2)*

```mermaid
graph TD
    A[Element type?] --> B{Primary action?}
    B -->|Yes| C[h-12 48px, full-width mobile]
    B -->|No| D{Secondary action?}
    D -->|Yes| E[h-10 or h-11 40-44px]
    D -->|No| F{Icon-only button?}
    F -->|Yes| G[w-11 h-11 44px square]
    F -->|No| H{Form input?}
    H -->|Yes| I[h-11 minimum 44px]
    H -->|No| J[min-h-[44px] with padding]
```

**Screen position adjustment:**

- Top of screen (hard to reach): ≥46px
- Center of screen (most precise): ≥27px minimum
- Bottom of screen (thumb zone): ≥44px

**Audit:** Are touch targets ≥44px on mobile?

### Mobile Content Display *(New in 6.2)*

```mermaid
graph TD
    A[Content length?] --> B{Title}
    B --> C[Mobile: line-clamp-2<br/>Desktop: line-clamp-none]
    A --> D{Description}
    D --> E[Mobile: line-clamp-2<br/>Desktop: full or line-clamp-3]
    A --> F{Tags/Items}
    F --> G[Mobile: 2-3 visible + count<br/>Desktop: all or 5 + count]
    A --> H{Actions}
    H --> I[Mobile: Stack primary/secondary<br/>Desktop: Horizontal row]
```

**Audit:** Is content appropriately truncated on mobile?

### Mobile Navigation *(New in 6.2)*

```mermaid
graph TD
    A[Number of nav items?] --> B{≤5 items}
    B -->|Yes| C[Show all as tabs/buttons<br/>Horizontal scroll if needed]
    B -->|No| D[Hamburger menu]
    D --> E{Nav complexity?}
    E -->|Simple list| F[Slide panel]
    E -->|Complex/nested| G[Full-screen overlay]
```

**Audit:** Can user see all nav without hamburger if ≤5 items?

## PATTERN CATALOG

### Three-Line Content Structure

**Title:** 14-16px, medium weight, high contrast (7:1)  
**Description:** 12-14px, regular, medium contrast (4.5:1), 3-4 line limit  
**Metadata:** 11-12px, regular, low contrast (3:1)

**Spacing:** 3-4px after title, 4px after description (8pt grid aligned)

**Mobile:** Single line preferred, max 2 lines for title *(Updated in 6.2)*

**Applies to:** Search results, list items, cards, table rows

### Grouped Container

```
SECTION HEADER (11px uppercase, gray-500)
┌─────────────────────────────────┐ ← Single border
│ Title            │    Metadata  │
│ Description      │              │
├─────────────────────────────────┤ ← Gradient divider
│ Title 2          │    Metadata  │
│ Description      │              │
└─────────────────────────────────┘
```

**When to group:** Items share type, user scans multiple, actions apply to group

**When NOT:** Items distinct, only 1-2 items, full-screen single item focus

**Divider options:**

- Subtle 1px line (default)
- Gradient fade (modern, softer visual separation)
- No divider if spacing sufficient (≥24px)

### Flexible Content Handling

**Accept:**

- Object with varying field names
- Plain text (single/multi-paragraph)
- Markdown (headings, bold, bullets, links, paragraphs)
- Null/undefined (show empty state)

**Field name alternatives (try in order):**

- Title: `title` → `headline` → `name` → `subject`
- Description: `description` → `summary` → `content` → `body`
- Date: `date` → `timestamp` → `published` → `created_at`

**Minimum markdown support (all content components):**

- Headings: `#` `##` `###`
- Bold: `**text**`
- Bullets: `- item`
- Paragraphs: Double newline

**Layout rules:**

- Headings: Single line with truncate + tooltip in cards
- Body: Natural wrap, leading-relaxed (1.5-1.6)
- Alignment: Top-align multi-column content
- Length: 3-4 lines visible, expand on demand

### Status Indication

**Text color only** (no background boxes):

- Success: `green-600`
- Warning: `amber-600`
- Info: `blue-600`
- Error: `red-600`
- Neutral: `gray-600`

**Exception:** Removable filter chips need subtle background (tap target)

**Why:** Background adds visual weight without information value

### Navigation States

**Top Nav (Fiverr-style):**

- Unselected: `gray-600` regular
- Hover: `gray-900` color only (150ms)
- Selected: `gray-900` medium + 2px bottom border
- Never: Background boxes or pills

**Side Nav (Google Classroom-style):**

- Unselected: `gray-700` + `hover:bg-gray-100`
- Selected: `bg-blue-50 text-blue-700` medium weight
- Icons: 18px same color as text
- Width: 224px fixed
- Padding: 12px horizontal, 8px vertical

**Section headers:** 11px uppercase, `gray-500`, tracking-wide

**Hierarchical (Folk-style):**

- Indent nested: 12px
- Chevron rotates 90° (150ms)
- Indicators: 8×8px squares
- Badges: End of row, minimal

**Dropdowns:**

- Width: 224px (matches side nav)
- Padding: 16px horizontal
- Dividers: 1px `gray-100`, 4px margin
- Hover: `bg-gray-50`
- Danger: Separated + `red-600` + `hover:bg-red-50`

### Element State Patterns

**HIDE:** Permanently unavailable (lacks permission)

- Don’t show in DOM
- Example: Admin features for non-admins

**DISABLE:** Contextually unavailable (user can fix)

- `bg-gray-200 text-gray-400 cursor-not-allowed`
- Optional helper text explaining condition
- Example: Save button until form valid

**MARK:** Available but requires action/upgrade

- Normal styling + badge
- “Coming soon” = amber badge
- “Pro” = purple badge, links to upgrade
- “Beta” = blue badge
- Example: Premium features, upcoming features

### Empty States

**Not:** “No results found”

**Instead:** Value-driven guidance

- Recent searches (clickable)
- Common searches
- Contextual suggestions
- Clear next action

**Pattern:** Optional icon → purpose statement → CTA

### Loading States

**<100ms:** No indicator (feels instant)  
**100ms-1s:** Spinner or pulse  
**1s-3s:** Skeleton screen (matches final layout)  
**>3s:** Progress bar (if determinate)

**Skeleton rules:**

- Match three-line pattern for lists
- Subtle shimmer (1.5s cycle)
- Max 5 skeleton items
- Progressive loading: header → content → secondary (50-100ms stagger)

## MOBILE PATTERNS *(New in 6.2)*

### Touch Targets

**WCAG 2.2 Requirements:**

- Level AA: ≥24×24px minimum
- Level AAA: ≥44×44px recommended
- Spacing: 4px minimum between adjacent targets under 24px

**Tailwind patterns:**

```html
<!-- Icon button: 44px target -->
<button class="w-11 h-11 flex items-center justify-center">
  <Icon size={16} />
</button>

<!-- Text button: proper height -->
<button class="h-11 px-4 text-sm font-medium">
  Action
</button>

<!-- Primary action: taller -->
<button class="h-12 w-full text-sm font-medium">
  Primary Action
</button>
```

### Mobile-First Classes

**Rule:** Base styles = mobile. Prefixes add complexity.

```html
<!-- ✓ Correct: mobile-first -->
<div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3">
<div class="flex flex-col md:flex-row">
<div class="text-base md:text-lg lg:text-xl">

<!-- ✗ Wrong: desktop-first -->
<div class="grid grid-cols-3 md:grid-cols-2 sm:grid-cols-1">
```

**Audit:** “Does the first class (no prefix) target mobile?”

### Content Truncation

```html
<!-- Title: 2 lines max on mobile -->
<h3 class="line-clamp-2 md:line-clamp-none">

<!-- Description: 2-3 lines max -->
<p class="line-clamp-2 md:line-clamp-3 lg:line-clamp-none">

<!-- Single line with truncate -->
<h4 class="truncate">
```

### Limited Items Display

```jsx
// Tags: show 2-3 + count
<div class="flex items-center gap-1.5">
  {tags.slice(0, 2).map(tag => <Tag>{tag}</Tag>)}
  {tags.length > 2 && (
    <span class="text-xs text-gray-500">+{tags.length - 2}</span>
  )}
</div>
```

### Action Stacking

```html
<!-- Mobile: stack vertically -->
<div class="flex flex-col gap-2">
  <!-- Primary: full width, tallest -->
  <button class="w-full h-12 bg-blue-600 text-white rounded-xl">
    Primary Action
  </button>
  
  <!-- Secondary: row, shorter -->
  <div class="flex gap-2">
    <button class="flex-1 h-10 bg-gray-100 rounded-lg">Secondary</button>
    <button class="flex-1 h-10 bg-gray-100 rounded-lg">Tertiary</button>
  </div>
</div>

<!-- Desktop: revert to row -->
<div class="flex flex-col md:flex-row gap-2">
  ...
</div>
```

### Input Sizing

**Rule:** Inputs ≥16px font size on mobile (prevents iOS auto-zoom).

```html
<input class="h-11 px-4 text-base rounded-xl" />
```

### Safe Area Padding

```html
<!-- Bottom padding for home indicator -->
<div class="pb-6"><!-- or pb-safe with plugin --></div>
```

```css
/* CSS fallback */
.safe-bottom {
  padding-bottom: max(1.5rem, env(safe-area-inset-bottom));
}
```

### Overflow Indicators

```html
<div class="relative">
  <div class="flex gap-2 overflow-x-auto scrollbar-hide">
    <!-- scrollable content -->
  </div>
  <!-- Fade hint -->
  <div class="absolute right-0 top-0 bottom-0 w-8 
              bg-gradient-to-l from-white to-transparent 
              pointer-events-none" />
</div>
```

### Active States

```html
<button class="bg-blue-600 active:bg-blue-700">
<button class="bg-gray-100 active:bg-gray-200">
```

### Scroll Margin

```css
/* For sticky headers with anchor links */
[id] {
  scroll-margin-top: 5rem;
}
```

### Always Ask Before

1. Individual borders on list items (violates grouping)
1. Full-width buttons for non-critical actions (violates Fitts)
1. Background boxes for status (violates signal-to-noise)
1. 2 icon colors in same context (violates cognitive load)
1. Technical jargon without confirmation (violates mental models)
1. Making non-functional elements look interactive (violates affordance)
1. Creating UI without real data source/API endpoint
1. Building forms without backend submission handler
1. Showing features with mock data that looks real
1. Desktop-first class ordering (should be mobile-first) *(6.2)*
1. Touch targets under 44px on mobile *(6.2)*
1. Showing all items when list exceeds 3-5 on mobile *(6.2)*

### Flexible Zones (User Can Override)

**Maintain principle, vary implementation:**

- Color values (maintain contrast ratios)
- Border radius (stay consistent)
- Font family (maintain size/weight ratios)
- Spacing values (stay on 8pt grid)

**Strict zones (audit as violations):**

- Group vs individual borders
- Button sizing by context
- Three-line hierarchy
- Touch targets ≥44px mobile
- Contrast ≥4.5:1 text, ≥3:1 large
- Content-chrome ≥70%
- Functional integrity (no fake buttons)
- Real data sources required
- Mobile-first class ordering *(6.2)*
- Content truncation on mobile *(6.2)*

### Content Handling Pattern

When building components:

```javascript
// Accept flexible input
function ContentCard({ data }) {
  // Handle different types
  if (typeof data === 'string') {
    return renderMarkdown(data);
  }
  
  // Extract with flexible field names
  const title = data.title || data.headline || data.name;
  const desc = data.description || data.summary || data.content;
  const meta = data.date || data.timestamp;
  
  // Render what exists, omit what doesn't
  return (
    <div>
      {title && <h3 className="font-medium">{title}</h3>}
      {desc && <p className="text-gray-600">{desc}</p>}
      {meta && <span className="text-gray-500 text-xs">{meta}</span>}
    </div>
  );
}
```

When generating content (in prompts):

```
Use markdown for rich formatting:
**Heading text**

Body paragraph with natural flow.

- Bullet point one
- Bullet point two

*Metadata info*
```

### Functional Integrity Implementation

**CRITICAL: Verify backend before building UI**


## ACCESSIBILITY (WCAG 2.2 AA)

### Non-Negotiable Minimums

|Requirement      |Minimum                        |Why                  |
|-----------------|-------------------------------|---------------------|
|Text contrast    |4.5:1 normal, 3:1 large        |Low vision legibility|
|Touch targets    |44×44px mobile, 24×24px desktop|Fitts’ Law for input |
|Focus indicators |Visible on all interactive     |Keyboard navigation  |
|Color + text/icon|Never color alone              |Colorblind support   |
|Motion respect   |Honor prefers-reduced-motion   |Vestibular disorder  |

### Quick Accessibility Checks

- [ ] Keyboard navigate entire UI?
- [ ] Focus visible on all interactive?
- [ ] Information clear without color?
- [ ] Touch targets meet minimums?
- [ ] Animations stop when user prefers reduced motion?

### Mobile Accessibility *(New in 6.2)*

- [ ] Skip link for keyboard users?
- [ ] Input font ≥16px (prevents iOS zoom)?
- [ ] Safe area padding for notch/home indicator?
- [ ] Active states on all tappable elements?

## 6.2.1 MOBILE ENHANCEMENT PATTERNS

> **Use when:** Building native-feeling mobile experiences, feed-based apps, or content-heavy mobile interfaces. These patterns add implementation complexity but significantly improve mobile UX.

### Swipe Actions

**When to use:**

- Feed cards that need quick save/dismiss
- List items with binary actions (archive/delete, approve/reject)
- Email or message interfaces

**Pattern:**

```jsx
<div className="relative overflow-hidden rounded-xl">
  {/* Action backgrounds */}
  <div className="absolute inset-y-0 left-0 w-20 bg-emerald-500 
                  flex items-center justify-center">
    <BookmarkIcon className="text-white" />
  </div>
  <div className="absolute inset-y-0 right-0 w-20 bg-red-500 
                  flex items-center justify-center">
    <TrashIcon className="text-white" />
  </div>
  
  {/* Swipeable card */}
  <div 
    className="bg-white relative transition-transform"
    style={{ transform: `translateX(${offset}px)` }}
  >
    {/* Card content */}
  </div>
</div>
```

**Rules:**

- Swipe right = positive action (save, approve, keep)
- Swipe left = negative action (delete, dismiss, archive)
- Show action icon in revealed background
- Threshold: 80px swipe to trigger action
- Provide haptic feedback on action trigger (native apps)

-----

### Bottom Sheet

**When to use:**

- Detail views on mobile (article expand, item details)
- Action menus with more than 3 options
- Forms that need context preservation

**Why bottom sheet over modal:**

- Thumb-friendly (action area at bottom)
- Preserves context (see content behind)
- Natural gesture to dismiss (swipe down)

**Pattern:**

```jsx
{showSheet && (
  <div className="fixed inset-0 z-50">
    {/* Backdrop */}
    <div className="absolute inset-0 bg-black/50" onClick={closeSheet} />
    
    {/* Sheet */}
    <div className="absolute bottom-0 left-0 right-0 
                    bg-white rounded-t-2xl max-h-[85vh] overflow-hidden">
      {/* Drag handle */}
      <div className="flex justify-center py-3">
        <div className="w-10 h-1 bg-gray-300 rounded-full" />
      </div>
      
      {/* Content */}
      <div className="px-5 pb-8 overflow-auto">
        {/* ... */}
        <div className="h-6" /> {/* Safe area */}
      </div>
    </div>
  </div>
)}
```

**Sizes:**

- Small (action menu): max-h-[40vh]
- Medium (details): max-h-[60vh]
- Large (forms): max-h-[85vh]

-----

### Pull to Refresh

**When to use:**

- Any scrollable list that updates
- Feed-based content
- Data that changes frequently

**Pattern:**

```jsx
{(isPulling || isRefreshing) && (
  <div className="absolute top-2 left-1/2 -translate-x-1/2 z-10 
                  bg-white shadow-lg rounded-full p-2">
    <RefreshCw className={`text-blue-600 ${isRefreshing ? 'animate-spin' : ''}`} />
  </div>
)}
```

**Rules:**

- Threshold: 60px pull distance to trigger
- Show spinner during refresh
- Return to top after refresh completes

-----

### Haptic Feedback (Native Apps)

**When to use:**

- Action completion (save, send, delete)
- Toggle switches
- Pull-to-refresh trigger
- Swipe action threshold reached

-----

### Gesture Hints

**When to use:**

- First-time user experience
- Non-obvious gestures (swipe, long-press)

**Pattern:**

```jsx
// Show once, persist dismissal
{!hasSeenHint && (
  <div className="absolute -top-8 left-1/2 -translate-x-1/2 
                  bg-gray-900 text-white text-xs px-2 py-1 rounded">
    Swipe to save
  </div>
)}
```

-----

### 6.2.1 Decision Tree

```
Building mobile interface?
│
├── Feed/list of cards?
│   ├── Need quick actions? → Add swipe gestures
│   ├── Cards expand to details? → Use bottom sheet
│   └── Content updates? → Add pull-to-refresh
│
├── Detail view from list?
│   └── Use bottom sheet (not modal)
│
├── Action menu with 3+ options?
│   └── Use bottom sheet (not dropdown)
│
└── Native app?
    └── Add haptic feedback on actions
```
